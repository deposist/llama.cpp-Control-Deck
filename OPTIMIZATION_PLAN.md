# План оптимизации llama.cpp Control Deck

## 1. Архитектура и структура кода

### 1.1 Разделение GUI и логики (приоритет: высокий)
- **Проблема**: `llama_cpp_gui.py` (~825 строк) смешивает UI-логику с бизнес-логикой.
- **Решение**: Выделить слой контроллера (controller) между GUI и менеджером.
  - `gui_controller.py` — обработка событий, валидация ввода, координация.
  - `llama_cpp_gui.py` — только отрисовка виджетов и привязка событий.
- **Выигрыш**: Тестируемость, возможность добавить CLI/web-интерфейс без дублирования.

### 1.2 Типизация и валидация конфигурации (приоритет: средний)
- **Проблема**: Конфигурация — вложенные dict без схемы. Ошибки обнаруживаются только в рантайме.
- **Решение**: Использовать `pydantic` (или `dataclasses` + ручная валидация) для моделей:
  - `ProfileConfig`, `InstanceConfig`, `ProxyConfig`, `AppConfig`.
- **Выигрыш**: Автодополнение в IDE, ранние ошибки, документация через типы.

### 1.3 Логирование вместо print (приоритет: средний)
- **Проблема**: Нет структурированного логирования в менеджере и GUI.
- **Решение**: Добавить `logging` с ротацией файлов и уровнями (DEBUG/INFO/WARNING/ERROR).
- **Выигрыш**: Диагностика проблем без открытия GUI.

## 2. Производительность

### 2.1 Оптимизация refresh_status (приоритет: высокий)
- **Проблема**: `refresh_status()` вызывается каждые 3 секунды и делает:
  - HTTP-запрос к каждому instance (`/v1/models`).
  - `psutil.net_connections()` для проверки портов.
  - Чтение логов с диска (`tail_file`).
  - Полная перестройка Treeview.
- **Решение**:
  - Кэшировать результаты `port_owner()` на 5-10 секунд.
  - Читать логи только при переключении на вкладку Logs.
  - Обновлять Treeview инкрементально (update values, а не delete+insert).
  - Использовать `threading` для HTTP health-check, чтобы не блокировать UI.
- **Выигрыш**: Снижение нагрузки на CPU и устранение подвисаний GUI.

### 2.2 Асинхронный health-check (приоритет: средний)
- **Проблема**: `http_json()` использует синхронный `urllib.request` с таймаутом 1-2 сек. При нескольких instances GUI может зависать.
- **Решение**: Выполнять health-check в фоновом потоке, результаты передавать через `queue.Queue` → `root.after()`.
- **Выигрыш**: GUI остается отзывчивым даже при недоступных серверах.

### 2.3 Ленивая загрузка устройств (приоритет: низкий)
- **Проблема**: `--list-devices` при старте может занимать до 10 секунд.
- **Решение**: Уже есть `--skip-device-refresh`. Дополнительно: кэшировать результат в `runtime/devices_cache.json` с TTL 60 секунд.

## 3. Надежность

### 3.1 Graceful shutdown (приоритет: высокий)
- **Проблема**: При закрытии GUI запущенные процессы остаются. Пользователь может забыть нажать Stop.
- **Решение**: Добавить `root.protocol("WM_DELETE_WINDOW", on_close)` с диалогом:
  - "Остановить все серверы перед выходом?" → Stop all + Quit / Quit without stopping / Cancel.
- **Выигрыш**: Меньше зависших процессов.

### 3.2 Обработка ошибок в Popen (приоритет: средний)
- **Проблема**: Если `llama-server` падает через 1-2 секунды после старта (а не мгновенно), состояние остается "running".
- **Решение**: После `start_instance()` запускать фоновую проверку через 3 и 10 секунд. Если процесс умер — обновить state и уведомить пользователя.
- **Выигрыш**: Точный статус без ручного обновления.

### 3.3 Атомарная запись state-файлов (приоритет: низкий)
- **Проблема**: Уже используется `os.replace()` для config.json, но state-файлы тоже.
- **Статус**: Реализовано корректно. Оставить как есть.

## 4. Ollama Proxy

### 4.1 Connection pooling (приоритет: высокий)
- **Проблема**: Каждый запрос создает новый `httpx.AsyncClient`. Это дорого по TCP-соединениям.
- **Решение**: Создать один `httpx.AsyncClient` на уровне приложения с `lifespan` event:
  ```python
  @asynccontextmanager
  async def lifespan(app):
      app.state.client = httpx.AsyncClient(timeout=SETTINGS["timeout"])
      yield
      await app.state.client.aclose()
  ```
- **Выигрыш**: Меньше latency, переиспользование TCP-соединений.

### 4.2 Retry и circuit breaker (приоритет: средний)
- **Проблема**: При временной недоступности llama-server proxy сразу возвращает 502.
- **Решение**: Добавить 1-2 retry с экспоненциальной задержкой (httpx + tenacity или ручной retry).
- **Выигрыш**: Устойчивость при кратковременных перезапусках сервера.

### 4.3 Поддержка /api/show и /api/ps (приоритет: низкий)
- **Проблема**: Некоторые Ollama-клиенты вызывают `/api/show` и `/api/ps`.
- **Решение**: Добавить заглушки, возвращающие минимальную информацию о модели.
- **Выигрыш**: Совместимость с большим числом клиентов.

## 5. UX и удобство

### 5.1 Индикация VRAM (приоритет: средний)
- **Проблема**: Пользователь не видит, сколько VRAM занято/свободно.
- **Решение**: Парсить вывод `nvidia-smi --query-gpu=memory.used,memory.total --format=csv` и показывать в статусной строке или вкладке GPU.
- **Выигрыш**: Пользователь понимает, поместится ли модель.

### 5.2 Предупреждение о конфликте портов при добавлении instance (приоритет: низкий)
- **Проблема**: Конфликт обнаруживается только при старте.
- **Решение**: Проверять порт при Apply и показывать предупреждение.

### 5.3 Темная тема (приоритет: низкий)
- **Проблема**: Tkinter по умолчанию выглядит устаревшим.
- **Решение**: Использовать `sv_ttk` или `ttkthemes` для современного вида.

  - `test_proxy.py` — httpx + TestClient для FastAPI endpoints.
